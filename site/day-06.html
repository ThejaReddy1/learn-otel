Which Signal Should I Use?

One of the most common early questions: do I use a log, metric, or trace here? This table helps you decide quickly:

Question you're answering Use Example
"What happened at 14:03:22?" Log Payment declined for user #4421
"How many requests per second?" Metric Counter: http_requests_total
"Why is this request slow?" Trace Span timeline shows DB query took 800ms
"Is latency getting worse over time?" Metric Histogram: request_duration_ms
"Which service in a chain is failing?" Trace Distributed trace spans STATUS=ERROR on checkout-svc
"Did the deploy cause errors?" All three Error rate spike (metric) + stack trace (log) + slow spans (trace)

Advanced Concept: Cardinality & Cost

Before you take an OTel setup into production, you must understand Cardinality. It is the #1 reason observability pipelines fail or result in massive bills.

⚠️ The Golden Rule of Metrics
NEVER put unbounded values (like User IDs, Request IDs, or Email Addresses) into a Metric label.

Low Cardinality (Safe for Metrics): HTTP Status Codes (200, 404, 500), HTTP Methods (GET, POST), Service Names. There are a fixed, small number of these.

High Cardinality (Dangerous for Metrics): User IDs (e.g., `user_id="847291"`). If you have 1 million users, your backend now has to store and track 1 million unique metric streams concurrently. This will crash your Prometheus server or result in a massive vendor bill.

So, where do you put the User ID?

You put High Cardinality data into Traces or Logs as attributes. Traces are discrete events (they just get appended), whereas Metrics are continuous aggregations.

Concept Quick Review

A span represents one unit of work — like "validate payment" or "query database". Spans have a name, start time, duration, attributes, and optionally a parent span.

Practiced in: Day 1

A trace is a group of related spans forming a tree. All spans in a trace share the same trace ID. The root span has no parent.

Practiced in: Day 1, Day 2

Resources describe the entity producing telemetry. Service name, version, and environment are common resource attributes. They stay consistent across all telemetry from the same source.

Practiced in: Day 1 — Resource Attributes

Context propagation moves trace context (trace ID, span ID) between services — typically via HTTP headers. Without it, each service creates an isolated trace.

Practiced in: Day 2 — API Instrumentation

A vendor-agnostic service that receives, processes, and exports telemetry. It decouples your application from backend specifics and centralizes telemetry pipeline management.

Practiced in: Day 3

OpenTelemetry Protocol — the standard wire format for transmitting telemetry. Available over gRPC (port 4317) and HTTP (port 4318).

Practiced in: Day 4

Test Your Understanding

These are OTCA-style questions. Try to answer before expanding each one.

Metric — specifically a Counter. You'd use meter.create_counter("checkout_total") and call .add(1, {"region": "us-east"}) on each checkout. Metrics answer "how many / how often."

Logs would record individual events but can't efficiently aggregate rates. Traces show request flow but don't naturally answer aggregate questions.

Context propagation — the client injects trace context into HTTP headers (inject(headers)), and each downstream service extracts it (extract(request.headers)) and continues the same trace.

Without propagation, each service produces a separate, disconnected trace. With it, all spans share the same trace_id and form a complete tree.

Practiced in: Day 2 — Distributed Tracing

SimpleSpanProcessor exports each span immediately when it ends — synchronous, blocks your app, good for dev/learning.

BatchSpanProcessor buffers spans and exports them in batches on a timer (default: every 5 seconds or when 512 spans accumulate) — async, much better for production.

Rule of thumb: SimpleSpanProcessor for learning, BatchSpanProcessor for production OTLP export.

The Collector centralizes telemetry pipeline management outside your application code. Key capabilities:

  • Receive telemetry from many services simultaneously
  • Transform, filter, or sample data before export
  • Route signals to multiple backends without changing app code
  • Add infrastructure metadata (k8s node name, pod labels, etc.)
  • Absorb traffic spikes and buffer data

Your app only needs to know one endpoint: the Collector's OTLP receiver. The Collector handles everything downstream.
