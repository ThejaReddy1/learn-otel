Module: Span Processors

A span processor sits between span creation and export. It decides when and how spans get sent to the exporter. This is a critical production decision.

🔧 Span Created
→
⚙️ Span Processor
→
📤 Exporter

SimpleSpanProcessor

Exports each span immediately as it completes. Great for learning and development, but adds latency to every operation in production.

python — immediate export
from opentelemetry.sdk.trace.export import SimpleSpanProcessor, ConsoleSpanExporter

exporter = ConsoleSpanExporter()
processor = SimpleSpanProcessor(exporter)
provider.add_span_processor(processor)

BatchSpanProcessor

Buffers spans and exports them in batches on a schedule. Much better for production — reduces network overhead and doesn't block your application while exporting.

python — batched export
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter

exporter = OTLPSpanExporter(endpoint="http://localhost:4318/v1/traces")
processor = BatchSpanProcessor(exporter)
provider.add_span_processor(processor)
✅ Production Rule of Thumb
Use SimpleSpanProcessor for local development and learning. Use BatchSpanProcessor for anything production-like. Batching is especially important when exporting over the network (OTLP).

Full Example — BatchSpanProcessor with OTLP

python — payment.py (batch + OTLP)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter

def configure_tracer():
    exporter = OTLPSpanExporter(endpoint="http://localhost:4318/v1/traces")
    span_processor = BatchSpanProcessor(exporter)
    provider = TracerProvider()
    provider.add_span_processor(span_processor)
    trace.set_tracer_provider(provider)
    return trace.get_tracer("payment.py", "0.1.0")

tracer = configure_tracer()

@tracer.start_as_current_span("Starting Payment")
def process_payment():
    print("processing payment")
    validate_card()
    charge_bank()

def validate_card():
    with tracer.start_as_current_span("Validating Card"):
        print("validating card")

def charge_bank():
    with tracer.start_as_current_span("Charging Bank"):
        print("charging bank")

if __name__ == "__main__":
    with tracer.start_as_current_span("Payment Service"):
        process_payment()

Module: Span Sampling

Sampling decides which traces to keep and which to discard. In high-traffic production systems, you can't afford to export every single trace. Sampling gives you control over this tradeoff.

🎯 The Core Tradeoff
More data = better debugging, higher cost (storage, network, compute)
Less data = lower cost, possible blind spots during incidents

Sampling Strategies

🎯
Always On
Keep everything. Best for learning, dev environments, or low-traffic services. The default.
🎲
TraceIdRatioBased
Keep a percentage of traces (e.g. 10%). Uses trace ID hash for consistent sampling decisions.
🚫
Always Off
Discard everything. Useful for testing overhead without export, or kill-switch scenarios.
🧠 Head-Based vs. Tail-Based Sampling
You will hear these two terms constantly. Here is the difference:
  • Head-Based (What we code below): The decision to keep or drop the trace is made at the very beginning of the request. Pros: Fast and cheap. Cons: You might drop a trace that eventually throws an error down the line.
  • Tail-Based: The Collector automatically holds the entire trace in memory until the request finishes, and only then decides to keep or drop it (e.g., "Keep 100% of errors, but only 5% of successes"). Pros: You never miss an error. Cons: Requires a massive, memory-heavy Collector.

Server with Sampling — charge.py

This server uses TraceIdRatioBased(0.1) — only 1 out of every 10 traces will be sampled and exported:

python — charge.py (sampled server)
from flask import Flask, request
from opentelemetry.sdk.resources import Resource
from opentelemetry import trace, context
from opentelemetry.propagate import extract
from opentelemetry.sdk.trace.sampling import TraceIdRatioBased
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor, ConsoleSpanExporter

def configure_tracer():
    exporter = ConsoleSpanExporter()
    span_processor = BatchSpanProcessor(exporter)
    resource = Resource.create({
        "service.name": "Charge service",
        "service.version": "0.4.0"
    })

    # Only sample 10% of traces
    sampler = TraceIdRatioBased(0.1)

    provider = TracerProvider(resource=resource, sampler=sampler)
    provider.add_span_processor(span_processor)
    trace.set_tracer_provider(provider)
    return trace.get_tracer("charge.py", "0.4.0")

tracer = configure_tracer()
app = Flask(__name__)

@app.before_request
def before_request_func():
    token = context.attach(extract(request.headers))
    request.environ["context_token"] = token

@app.teardown_request
def teardown_request_function(err):
    token = request.environ.get("context_token", None)
    if token:
        context.detach(token)

@app.route("/charge")
@tracer.start_as_current_span("Charge Account", kind=trace.SpanKind.SERVER)
def charge():
    span = trace.get_current_span()
    span.set_attributes({
        "HTTP_METHOD": request.method,
        "CLIENT_IP": request.remote_addr,
        "HTTP_PATH": request.path
    })
    span.add_event("User has insufficient Funds, only has $5")
    return "Charging Users Bank Account"

if __name__ == "__main__":
    app.run(debug=True)
⚙️ Equivalent Environment Variables
You don't have to code your sampler into the app. Using standard OpenTelemetry environment variables achieves the exact same result automatically (very important for the OTCA exam):

OTEL_TRACES_SAMPLER=traceidratio
OTEL_TRACES_SAMPLER_ARG=0.1
⚠️ Sampling is probabilistic
With TraceIdRatioBased(0.1), roughly 10% of traces are kept. Run the client multiple times — some requests produce trace output, others don't. This is expected behavior, not a bug.
💡 Distributed Systems: Use ParentBased
TraceIdRatioBased makes independent sampling decisions on each service. In a distributed system, a child span might sample while the parent did not — breaking trace continuity.

The recommended production pattern wraps it with ParentBased, which respects the parent's sampling decision:
sampler = ParentBased(root=TraceIdRatioBased(0.1))

Or use the convenience alias: ParentBasedTraceIdRatio(0.1)

Client with Context Propagation — payment.py

The client sends requests with trace context injected into headers. It also demonstrates adding custom span attributes and recording exceptions:

python — payment.py (client with propagation)
import requests
from opentelemetry.sdk.resources import Resource
from opentelemetry import trace
from opentelemetry.propagate import inject
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor, ConsoleSpanExporter

def configure_tracer():
    exporter = ConsoleSpanExporter()
    span_processor = BatchSpanProcessor(exporter)
    resource = Resource.create({
        "service.name": "payment_service",
        "service.version": "0.2.0"
    })
    provider = TracerProvider(resource=resource)
    provider.add_span_processor(span_processor)
    trace.set_tracer_provider(provider)
    return trace.get_tracer("payment.py", "0.1.0")

tracer = configure_tracer()

@tracer.start_as_current_span("Starting Payment")
def process_payment():
    span = trace.get_current_span()
    span.set_attributes({
        "user": "john",
        "account_number": "7372349",
        "bank": "SBI"
    })
    print("processing payment")
    validate_card()
    charge_bank()

@tracer.start_as_current_span("Validating Card")
def validate_card():
    print("validating card")

@tracer.start_as_current_span("Charge Bank")
def charge_bank():
    print("charging bank")
    with tracer.start_as_current_span("Request to Charge API", kind=trace.SpanKind.CLIENT) as span:
        try:
            url = "http://127.0.0.1:5000/charge"
            # NOTE: In production, use standard OTel Semantic Conventions instead of custom keys:
            # "http.request.method" instead of "HTTP_METHOD"
            # "url.full" instead of "HTTP_URL"
            span.set_attributes({"HTTP_METHOD": "GET", "HTTP_URL": url})
            headers = {}
            inject(headers)  # Inject trace context into HTTP headers
            span.add_event("Sending Request")
            resp = requests.get(url, headers=headers)
            span.add_event("Request Sent", attributes={"url": url})
            span.set_attribute("HTTP_STATUS_CODE", resp.status_code)
        except Exception as err:
            span.record_exception(err)  # OTel records exceptions automatically

if __name__ == "__main__":
    with tracer.start_as_current_span("Payment Service"):
        process_payment()
🧠 inject() vs extract()
inject(headers) — Client-side. Puts trace ID and span ID into outgoing HTTP headers.
extract(request.headers) — Server-side. Reads trace context from incoming HTTP headers.

Together, they create distributed traces across service boundaries.
