Day 0 — Orientation

Foundations Before You Instrument Anything

Before creating spans, starting the Collector, or exporting telemetry, build a clean mental model for observability. Day 0 is about understanding what logs, metrics, and traces actually do — and where OpenTelemetry fits.

⏱ ~15 min reading 🟢 Beginner

Why Day 0 Matters

Many beginners jump directly into traces, metrics, or the Collector and end up following commands without understanding the system behind them. Day 0 fixes that.

By the end of this day, you should be able to answer:

  • What problem does observability solve?
  • How are logs, metrics, and traces different?
  • Where does OpenTelemetry fit in the observability stack?
  • What does the Collector do?
  • What ideas show up repeatedly in OTCA learning?

Observability In Plain Language

Observability is the ability to understand what a system is doing from the signals it produces.

In practice, that means answering questions like:

  • Why is this request slow?
  • Which service is failing?
  • Did latency increase after a deploy?
  • Are errors isolated or system-wide?
  • Can I explain what happened without guessing?
💡 Key Insight
Observability is not just dashboards. It is a way to investigate unknown problems using telemetry.

The Three Core Signals

Logs

Logs are event records. Use logs when you want discrete details such as:

  • An exception message
  • A user action
  • A deployment event
  • A business event like "payment failed"
💭 Think
"Something happened."

Metrics

Metrics are numeric measurements over time. Use metrics when you want:

  • Request counts
  • Error rates
  • CPU and memory trends
  • Latency percentiles
💭 Think
"How much, how often, how fast?"

Traces

Traces show the lifecycle of a request or operation across one or more services. Use traces when you want:

  • Request flow across services
  • Timing by step
  • Parent-child execution relationships
  • Context during debugging
💭 Think
"Where did the time go, and what happened along the way?"

OpenTelemetry In Plain Language

OpenTelemetry is the open standard and tooling ecosystem used to generate, collect, process, and export telemetry. It helps you:

  • Instrument applications
  • Create traces, metrics, and logs
  • Standardize telemetry pipelines
  • Export data to many backends
⚠️ Important
OpenTelemetry is not the backend where you analyze data. It is the layer that helps produce and move telemetry.

The Mental Model & Architecture

Before writing code, you need a mental map of how data actually moves in a production system. OpenTelemetry architecture is typically divided into three layers:

🖥️ Edge Layer
The App / First Mile
• Generates traces/metrics
• SDK or Auto-Instrumentation
• Sends data via OTLP
→
📦 Gateway Layer
The Collector
• Center of Trust
• Filtering & Batching
• Tail Sampling (Decisions)
→
☁️ Egress Layer
The Backends
• Observability Tools
• Grafana, Jaeger, etc.
• Final Storage & Alerts
💡 Why a Gateway Layer?
You *could* send data directly from your App (Edge) straight to your Backend (Egress). But in production, you use a Collector Gateway. Why? Because the Gateway provides a robust buffer against network outages, allows you to strip PII before it leaves your network, and lets you sample (drop) traces to save on massive storage costs.

Key Terms You Should Know

Span

A span represents one unit of work. Examples: "validate payment", "query database", "call shipping service".

Trace

A trace is a group of related spans that represent one full request or workflow.

Resource

A resource describes the entity producing telemetry — service name, version, environment, host metadata.

Attributes

Attributes are key-value details attached to telemetry — HTTP method, route, user region, database name.

Context Propagation

Context propagation is how trace context moves between services so multiple spans become one connected trace. Think of a trace like a relay race, and the Trace ID is the baton.

When Service A makes an HTTP call to Service B, it literally injects a hidden HTTP header (e.g., traceparent: 00-4bf92...) into the request. Service B reads that header, takes the baton, and continues the exact same trace.

Exporter

An exporter sends telemetry out of the app or Collector — OTLP exporter, Prometheus exporter, vendor-specific exporters.

Collector

The Collector is a separate service that receives, processes, and exports telemetry. It centralizes pipelines and keeps application code cleaner.

App vs Collector Responsibilities

🖥️
Application Side
  • Creating spans, metrics, and logs
  • Adding business and request context
  • Exporting to OTLP or another receiver
📦
Collector Side
  • Receiving telemetry from many apps
  • Batching or processing data
  • Routing to destinations
  • Reducing backend coupling

OTCA-Oriented Understanding

If you are learning with OTCA in mind, Day 0 should make these idea groups feel clear:

  • Telemetry signals: logs, metrics, traces
  • Instrumentation basics: manual, library-based, auto-instrumentation
  • Resources and attributes
  • Context propagation
  • Collector roles: receivers, processors, exporters, pipelines
  • OTLP as a common transport format
📝 Note
You do not need to memorize everything now. You only need a clean mental model so the later hands-on examples make sense.

Glossary of Acronyms

OpenTelemetry is filled with acronyms. Here is your cheat sheet:

  • OTel: OpenTelemetry (the open-source project itself).
  • OTLP: OpenTelemetry Protocol (the language the data speaks over the network).
  • OTCA: OpenTelemetry Certified Associate (the official Linux Foundation exam).
  • W3C TraceContext: The official internet standard for the hidden HTTP headers used in context propagation.
