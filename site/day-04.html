Module: OTLP Exporting

OTLP (OpenTelemetry Protocol) is the standard wire format for sending telemetry. Until now, you've been using ConsoleSpanExporter to see traces in the terminal. In production, you use OTLP to send data to the Collector.

💡 OTLP Endpoints
gRPC: localhost:4317 — binary protocol, most efficient
HTTP: localhost:4318/v1/traces — works through proxies and firewalls

Our examples use HTTP because it's easier to debug and requires fewer dependencies.

The Code

This example sends both traces and logs to the Collector via OTLP. Notice how spans and logs are created in the same flow — the trace context is automatically attached to log records:

python — main.py (OTLP export)
import logging
from opentelemetry.sdk._logs import LoggerProvider, LoggingHandler
from opentelemetry._logs import set_logger_provider, get_logger
from opentelemetry.sdk._logs.export import BatchLogRecordProcessor

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import ConsoleSpanExporter, SimpleSpanProcessor

# OTLP exporter for logs — sends to the Collector
from opentelemetry.exporter.otlp.proto.http._log_exporter import OTLPLogExporter

# --- Trace setup (console for visibility) ---
exporter = ConsoleSpanExporter()
span_processor = SimpleSpanProcessor(exporter)
provider = TracerProvider()
provider.add_span_processor(span_processor=span_processor)
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("flight-booking", "0.1.2")

# --- Log setup (OTLP to Collector) ---
logger_provider = LoggerProvider()
otlp_exporter = OTLPLogExporter(endpoint="http://localhost:4318/v1/logs")
processor = BatchLogRecordProcessor(exporter=otlp_exporter, export_timeout_millis=5000)
logger_provider.add_log_record_processor(processor)
set_logger_provider(logger_provider=logger_provider)

# Bridge Python logging to OTel
handler = LoggingHandler(level=logging.NOTSET, logger_provider=logger_provider)
logging.basicConfig(handlers=[handler], level=logging.NOTSET)

@tracer.start_as_current_span("Book Flight")
def book_flight():
    logging.info("Starting the booking flight process")
    search_flights()
    reserve_flight()

@tracer.start_as_current_span("Search Flight")
def search_flights():
    logging.info("Searching for flights")

@tracer.start_as_current_span("Reserve Flight")
def reserve_flight():
    logging.warning("Reserving flight")

if __name__ == "__main__":
    book_flight()
🧠 What's happening here?
Two exporters, two destinations:
• Traces → Console (so you can see them locally)
• Logs → OTLP endpoint (sent to the Collector)

In production, you'd typically send everything via OTLP to the Collector, and the Collector would route it to your backends (Jaeger, Grafana, etc.).

Module: Trace-Log Correlation

When traces and logs are produced in the same flow, OpenTelemetry automatically attaches trace context to log records. This means you can click on a log entry in your backend and jump directly to the trace — or vice versa.

The Code

python — main.py (trace-log correlation)
import logging
from opentelemetry.sdk._logs import LoggerProvider, LoggingHandler
from opentelemetry._logs import set_logger_provider, get_logger
from opentelemetry.sdk._logs.export import ConsoleLogExporter, BatchLogRecordProcessor

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import ConsoleSpanExporter, SimpleSpanProcessor

# Both traces and logs export to console — easy to compare
trace_exporter = ConsoleSpanExporter()
trace.set_tracer_provider(TracerProvider())
trace.get_tracer_provider().add_span_processor(SimpleSpanProcessor(trace_exporter))
tracer = trace.get_tracer("flight-booking", "0.1.2")

logger_provider = LoggerProvider()
log_exporter = ConsoleLogExporter()
processor = BatchLogRecordProcessor(exporter=log_exporter, export_timeout_millis=5000)
logger_provider.add_log_record_processor(processor)
set_logger_provider(logger_provider=logger_provider)

handler = LoggingHandler(level=logging.NOTSET, logger_provider=logger_provider)
logging.basicConfig(handlers=[handler], level=logging.NOTSET)

@tracer.start_as_current_span("Book Flight")
def book_flight():
    logging.info("Starting the booking flight process")
    search_flights()
    reserve_flight()

@tracer.start_as_current_span("Search Flight")
def search_flights():
    logging.info("Searching for flights")

@tracer.start_as_current_span("Reserve Flight")
def reserve_flight():
    logging.warning("Reserving flight")

if __name__ == "__main__":
    book_flight()
🔗 The Correlation Magic
Look at the output — both spans and log records share the same trace_id and span_id. This is trace-log correlation in action. In a real backend (Grafana, Jaeger), clicking a log entry takes you directly to the trace, and clicking a span shows you the logs emitted during that operation.

Module: Prometheus Exporter

Prometheus works differently from OTLP. Instead of your app pushing data, Prometheus pulls (scrapes) metrics from an HTTP endpoint your app exposes. This is called the "pull model."

🖥️ Your App
Exposes /metrics
←
📊 Prometheus
Scrapes every N seconds
→
📈 Dashboard
Grafana, etc.

The Code

This is a Flask app with full Prometheus metrics — counters, histograms, gauges, and up-down counters all in one service:

python — main.py (Prometheus exporter)
import random, time, os, psutil
from flask import Flask, request
from opentelemetry.metrics import set_meter_provider, get_meter_provider, Observation
from opentelemetry.sdk.metrics.view import ExplicitBucketHistogramAggregation, View
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.resources import Resource
from opentelemetry.exporter.prometheus import PrometheusMetricReader
from prometheus_client import start_http_server

def configure_meter():
    # Start a separate HTTP server for Prometheus to scrape
    start_http_server(port=8000, addr="localhost")
    reader = PrometheusMetricReader()

    # Custom histogram buckets for request duration
    buckets = [0.0, 20, 30, 60, 100, 250, 2000]
    view = View(
        instrument_name="total_request_duration",
        aggregation=ExplicitBucketHistogramAggregation(boundaries=buckets)
    )

    provider = MeterProvider(
        metric_readers=[reader],
        resource=Resource.create(),
        views=[view]
    )
    set_meter_provider(provider)
    return get_meter_provider().get_meter(name="shopping-app", version="0.1.2")

meter = configure_meter()
app = Flask(__name__)

# Counter — total requests by route and method
requests_counter = meter.create_counter(
    "http_requests_total",
    description="Total number of requests processed",
    unit="1"
)

# Up-Down Counter — concurrent requests in progress
concurrent_requests = meter.create_up_down_counter(
    "concurrent_requests",
    description="Number of requests currently in progress",
    unit="1"
)

# Observable Gauge — memory usage via callback
process = psutil.Process(os.getpid())
def memory_usage_callback(options):
    mem_info = process.memory_info()
    return [
        Observation(mem_info.rss, {"type": "rss"}),
        Observation(mem_info.vms, {"type": "vms"})
    ]

meter.create_observable_gauge(
    "application_memory_usage",
    description="Memory usage of application",
    unit="By",
    callbacks=[memory_usage_callback]
)

# Histogram — request duration
total_request_duration = meter.create_histogram(
    "total_request_duration",
    description="Request Duration",
    unit="ms"
)

@app.before_request
def before_request():
    request.start_time = time.time()
    concurrent_requests.add(1)

@app.after_request
def after_request(response):
    request_latency = (time.time() - request.start_time) * 100
    concurrent_requests.add(-1)
    total_request_duration.record(request_latency)
    return response

@app.get("/products")
def get_products():
    requests_counter.add(1, {"route": "/products", "method": request.method})
    time.sleep(random.uniform(0.05, 1.5))
    return "Get All Products"

@app.get("/cart")
def get_cart():
    requests_counter.add(1, {"route": "/cart", "method": request.method})
    time.sleep(random.uniform(0.05, 1.5))
    return "Get Cart"

if __name__ == "__main__":
    app.run()
🧠 All four metric types in one app
Counter → http_requests_total — always goes up
Up-Down Counter → concurrent_requests — goes up on request start, down on response
Observable Gauge → application_memory_usage — read via callback when scraped
Histogram → total_request_duration — tracks latency distribution

  • For Prometheus output, keep the app running while you visit http://localhost:8000/metrics.
  • If export errors mention connection refusal, the Collector is not running or the endpoint is wrong.
  • If you only see spans or only logs in correlation, verify that both providers are configured before the functions run.
