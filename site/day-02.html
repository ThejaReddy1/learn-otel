Day 2 — Practical

Instrumentation

Instrumentation is where OpenTelemetry becomes practical. You stop thinking about abstract telemetry and start tracing real HTTP calls, logging meaningful events, and counting what your application is doing.

⏱ ~60 min hands-on 🟦 Beginner — Intermediate

Learning Outcomes

  • Instrument Flask services and client calls
  • Understand distributed tracing and context propagation
  • Capture logs and metrics alongside traces
  • Compare manual, library-based, and auto-instrumentation approaches

Prerequisites

py -m pip install -r .\shared-resources\python\requirements.txt
py -m pip install -r .\shared-resources\python\requirements-auto.txt

API Instrumentation

📚
Library Instrumentation
Automatic spans from instrumented libraries like requests and Flask.
⚡
Auto-Instrumentation

The Three Instrumentation Approaches

Before diving into the modules, understand the three ways you can instrument an application:

🔧 Manual
Full control, you write every span
|
📚 Library
Pre-built spans from instrumented libs
|
⚡ Auto
Zero code changes, wraps your app
💡 When to use which?
Manual — when you need custom business spans (e.g. "validate payment").
Library — when you want automatic spans for HTTP, database, or framework calls.
Auto — when you want telemetry without changing application code at all.
🏆 The "Golden Path" Cheat Sheet
Getting stuck on configuration is the #1 reason beginners give up on OTel. Here is the industry-standard "Golden Path" for Python. Always start here:

1. Install the CLI & Auto-instrumentation bootstrap: 
pip install opentelemetry-distro opentelemetry-exporter-otlp
opentelemetry-bootstrap -a install
2. Run your app wrapped in the OTel CLI (Zero Code Changes!): 
export OTEL_SERVICE_NAME="my-awesome-python-app"
export OTEL_TRACES_EXPORTER="otlp"
export OTEL_EXPORTER_OTLP_ENDPOINT="http://localhost:4317"

opentelemetry-instrument python my_app.py
Only drop down into manual library instrumentation if the Auto-instrumentation doesn't capture the specific business logic you need!

Module: API Instrumentation

These examples build a trace story step by step: instrument a Flask server, connect two services, propagate context, record errors, and observe status-related behavior.

Server Setup — Your First Instrumented API

This is a Flask server with manual tracing. The charge.py server creates a span for every incoming request, extracts trace context from headers, and attaches useful attributes:

python — charge.py (server)
from flask import Flask, request
from opentelemetry.sdk.resources import Resource
from opentelemetry import trace, context
from opentelemetry.propagate import extract
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor, ConsoleSpanExporter

def configure_tracer():
    exporter = ConsoleSpanExporter()
    span_processor = BatchSpanProcessor(exporter)
    resource = Resource.create({
        "service.name": "Charge service",
        "service.version": "0.4.0"
    })
    provider = TracerProvider(resource=resource)
    provider.add_span_processor(span_processor)
    trace.set_tracer_provider(provider)
    return trace.get_tracer("charge.py", "0.4.0")

tracer = configure_tracer()
app = Flask(__name__)

# Extract incoming trace context from HTTP headers
@app.before_request
def before_request_func():
    token = context.attach(extract(request.headers))
    request.environ["context_token"] = token

@app.teardown_request
def teardown_request_function(err):
    token = request.environ.get("context_token", None)
    if token:
        context.detach(token)

@app.route("/charge")
@tracer.start_as_current_span("Charge Account", kind=trace.SpanKind.SERVER)
def charge():
    span = trace.get_current_span()
    span.set_attributes({
        "HTTP_METHOD": request.method,
        "CLIENT_IP": request.remote_addr,
        "HTTP_PATH": request.path
    })
    span.add_event("User has insufficient Funds, only has $5")
    return "Charging Users Bank Account"

if __name__ == "__main__":
    app.run(debug=True)
🧠 What's happening here?
context.attach(extract(...)) pulls trace context from incoming HTTP headers. This is how the server "joins" an existing trace started by a client.

span.set_attributes(...) attaches metadata to the span. Later, when you view this trace, you'll see the HTTP method, client IP, and path right alongside the timing data.

span.add_event(...) records a timestamped event inside the span — like a structured log entry within the trace.
🏷️ OTel Semantic Conventions for HTTP Attributes
The example above uses custom uppercase attribute names like "HTTP_METHOD" and "CLIENT_IP" for clarity. In production, use the official OTel semantic conventions:

"http.request.method" instead of "HTTP_METHOD"
"url.full" instead of "HTTP_URL"
"client.address" instead of "CLIENT_IP"

Semantic conventions ensure your attributes match what observability backends expect, enabling automatic dashboards and alerts. Read the HTTP semantic conventions ↗

Module: Library Instrumentation

Instead of manually creating spans for every HTTP call, you can let a library instrumentor do it for you. The RequestsInstrumentor automatically creates client spans for every outgoing HTTP request made with the requests library.

The Client — payment.py

python — payment.py (library-instrumented client)
import requests
from opentelemetry import trace
from opentelemetry.sdk.resources import Resource
from opentelemetry.instrumentation.requests import RequestsInstrumentor
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import SimpleSpanProcessor, ConsoleSpanExporter

resources = Resource.create({
    "service.name": "Casher API",
    "service.version": "0.3.0"
})

trace.set_tracer_provider(TracerProvider(resource=resources))
trace.get_tracer_provider().add_span_processor(
    SimpleSpanProcessor(ConsoleSpanExporter())
)

# This one line instruments ALL outgoing HTTP requests
RequestsInstrumentor().instrument()

def charge_bank():
    print("charging bank")
    try:
        url = "http://127.0.0.1:5000/charge"
        resp = requests.get(url)
    except Exception as err:
        print(err)

def main():
    print("Calling bank charger...")
    charge_bank()
    print("Done! Charging bank done...")

if __name__ == "__main__":
    main()
🧠 What's the difference from manual?
With manual instrumentation (Day 1), you wrote tracer.start_as_current_span("...") around every operation.

With library instrumentation, you call RequestsInstrumentor().instrument() once, and every HTTP call made with requests.get() automatically gets a span with HTTP method, URL, status code, and timing — no extra code needed per request.

Module: Auto-Instrumentation

Auto-instrumentation is the zero-code approach. You don't add any OpenTelemetry imports or tracer setup to your application. Instead, you wrap the app at startup using a command-line tool.

The Application — No OTel Code!

python — products.py (plain Flask, no OTel code)
from flask import Flask

app = Flask(__name__)

@app.route("/")
def charge():
    return "Here are all of the available products"

if __name__ == "__main__":
    app.run(debug=False, use_reloader=False)
⚠️ Notice!
There are zero OpenTelemetry imports in this file. The app is completely unaware of tracing. All instrumentation happens externally at startup.

Step-by-Step

1

Install instrumentors automatically

This command scans your installed packages and installs matching OpenTelemetry instrumentors:

powershell
opentelemetry-bootstrap -a install
2

Start the app with auto-instrumentation

powershell
opentelemetry-instrument --traces_exporter console --metrics_exporter console --service_name products py .\otel-learning\day-02-instrumentation\auto-instrumentation\products.py
3

Send a request

Open http://127.0.0.1:5000/ in your browser and check the terminal for trace output.

💡 CLI vs Environment Variables
You can configure auto-instrumentation via CLI flags or environment variables:

--traces_exporter → OTEL_TRACES_EXPORTER
--metrics_exporter → OTEL_METRICS_EXPORTER
--service_name → OTEL_SERVICE_NAME
--exporter_otlp_endpoint → OTEL_EXPORTER_OTLP_ENDPOINT

Note: With env vars, the /v1/traces path is not auto-appended — you must configure it manually.

Module: Metrics

Metrics answer "how much, how often, how fast?" OpenTelemetry provides four core metric instruments. Each is designed for a different measurement pattern:

🔢
Counter
Always goes up — track total requests, orders placed, errors. counter.add(1)
↕️
Up-Down Counter
Can go up or down — track queue sizes, concurrent connections. counter.add(-1)
📊
Histogram
Measures distributions — request duration, payload sizes. histogram.record(42.5)
📡
Observable Gauge
Snapshot values via callbacks — memory usage, CPU, temperature. Reads value periodically.

⏱️ No output appearing?
Metrics are exported on an interval (typically every 5-10 seconds). Wait a moment — unlike spans which export as they complete, metrics batch on a timer.

Module: Logging Basics

OpenTelemetry can capture your existing Python logs and export them as structured telemetry. This means your familiar logging.info("...") calls can be enriched with trace context and sent to observability backends.

The Code

python — main.py (OTel logging)
import logging
from opentelemetry.sdk._logs import LoggerProvider, LoggingHandler
from opentelemetry._logs import set_logger_provider, get_logger
from opentelemetry.sdk._logs.export import ConsoleLogExporter, BatchLogRecordProcessor

# 1. Create a logger provider (like TracerProvider, but for logs)
logger_provider = LoggerProvider()

# 2. Create an exporter (console for learning, OTLP for production)
exporter = ConsoleLogExporter()

# 3. Create a processor that batches log records
processor = BatchLogRecordProcessor(
    exporter=exporter,
    export_timeout_millis=5000
)

# 4. Wire it all together
logger_provider.add_log_record_processor(processor)
set_logger_provider(logger_provider=logger_provider)

# 5. Bridge Python's standard logging to OTel
handler = LoggingHandler(level=logging.NOTSET, logger_provider=logger_provider)
logging.basicConfig(handlers=[handler], level=logging.NOTSET)

# Now standard Python logs flow through OpenTelemetry!
def book_flight():
    logging.info("Starting the booking flight process")
    search_flights()
    reserve_flight()

def search_flights():
    logging.info("Searching for flights")

def reserve_flight():
    logging.warning("Reserving flight")

if __name__ == "__main__":
    book_flight()
💡 The Pattern: Provider → Processor → Exporter
Notice the same pattern as tracing?
Traces: TracerProvider → SpanProcessor → SpanExporter
Logs: LoggerProvider → LogRecordProcessor → LogExporter
Metrics: MeterProvider → MetricReader → MetricExporter

OTel uses the same architecture for all three signals. Learn it once, apply everywhere.
