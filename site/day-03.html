Day 3 — Deep Dive

Collector Deep Dive

Until now, your apps exported telemetry straight to the console. The Collector adds a production-grade control point between your app and your backends — one place to receive, process, filter, and route all your signals.

⏱ ~45 min hands-on 💓 Intermediate

Learning Outcomes

  • Explain why the Collector exists and what problem it solves
  • Understand receivers, processors, exporters, and service pipelines
  • Read and write a Collector config YAML
  • Start the Collector with Docker and verify telemetry flows through it

Why the Collector?

A common beginner question: "If I can just send OTLP data directly from my Python code to Jaeger, why do I need to set up this complex Collector middleware?"

While direct-to-backend works for simple tutorials, in production the Collector is vital because it:

  • Offloads work: It handles retries, batching, and encryption so your app doesn't waste CPU/memory doing it.
  • Abstracts vendors: Your app only talks to the Collector. If you switch from Datadog to Grafana tomorrow, you only change the Collector config, not your app code.
  • Secures credentials: API keys for your observability SaaS live in the Collector, not inside your application's environment variables.

💡 One app, many backends
Your app always speaks OTLP to the Collector. The Collector decides where to send data — Jaeger for traces, Prometheus for metrics, Loki for logs. You never need to change your application code when switching backends.

Collector Architecture

Every Collector config is built from four section types. A pipeline is just a named combination of receivers → processors → exporters for one signal type.

📥
Receivers
Listen for incoming telemetry. OTLP receiver accepts gRPC on port 4317 and HTTP on port 4318.
⚙️
Processors
Transform, filter, or batch telemetry before export. The batch processor groups spans to reduce network calls.
📤
Exporters
Send telemetry to a backend. The debug exporter prints to the Collector's own stdout — great for learning.
🔗
Service Pipelines
Wire receivers → processors → exporters together per signal type: traces, metrics, and logs each get their own pipeline.

The Config Files

The shared collector directory contains two configs. Start with the minimal one to understand the bare structure, then move to the debug config which adds the batch processor:

Minimal Config — collector-config-minimal.yaml

The simplest possible Collector: one OTLP receiver, no processors, and the debug exporter printing every signal to stdout. Good for first-time verification that telemetry arrives.

yaml — collector-config-minimal.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317   # gRPC — binary protocol
      http:
        endpoint: 0.0.0.0:4318   # HTTP — works through proxies

processors:
  # None — data flows straight through

exporters:
  debug:
    verbosity: detailed           # Prints full telemetry to Collector stdout

service:
  pipelines:
    traces:                       # One pipeline per signal type
      receivers: [otlp]
      exporters: [debug]
    metrics:
      receivers: [otlp]
      exporters: [debug]
    logs:
      receivers: [otlp]
      exporters: [debug]

Debug Config — collector-config-debug.yaml

Adds the batch processor to each pipeline. Instead of forwarding each span individually, the Collector waits to accumulate a batch and sends them together. This is much more efficient for production workloads.

yaml — collector-config-debug.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:                          # Buffers and batches spans before export

exporters:
  debug:
    verbosity: detailed

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]         # Added: spans are batched here
      exporters: [debug]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]
🧠 Reading the config: top-down vs service section
The top of the config defines available components (what is declared). The service.pipelines section defines what is actually active. You can declare a receiver or exporter and never wire it into a pipeline — it simply won't run. This is useful for staging config changes.

Docker Setup

The Collector runs as a Docker container. The compose file in shared-resources/collector/ maps the standard OTLP ports and mounts the config file:

yaml — docker-compose.yaml
services:
  otel-collector:
    image: otel/opentelemetry-collector-contrib:latest
    command: ["--config=/etc/otel-collector-config.yaml"]
    volumes:
      - ./collector-config-debug.yaml:/etc/otel-collector-config.yaml
    ports:
      - "4317:4317"   # gRPC
      - "4318:4318"   # HTTP

Step-by-Step

1

Start the Collector

Run from the repository root. Docker Desktop must be running.

powershell
docker compose -f .\shared-resources\collector\docker-compose.yaml up -d
2

Verify the container is running

powershell
docker ps

You should see an otel-collector container with ports 4317 and 4318 mapped.

3

Send telemetry to the Collector

Re-run any OTLP example from Day 4 or the shopping cart from Day 7 — both target localhost:4318 which the Collector is now listening on:

powershell
py .\otel-learning\day-04-exporters\otlp-exporting\main.py
4

Inspect the Collector logs

The debug exporter prints full telemetry to stdout. You'll see span names, attributes, trace IDs, and timestamps:

powershell
docker logs otel-collector

Config Section Anatomy

Every component in the config follows the same pattern: declare it in its section, then reference it in a pipeline. Here's what each receiver and exporter key means:

🔌
receivers.otlp
Activates the OTLP receiver. Sub-keys grpc and http define the protocols and bind addresses.
🔄
processors.batch
Groups spans into batches for efficient export. Default: sends when batch reaches 512 items or every 5 seconds (5000ms), whichever comes first.
🖨️
exporters.debug
Prints telemetry to the Collector's own log output. verbosity: detailed shows full span data including all attributes.
📋
service.pipelines
One section per signal type. Each pipeline names which receivers, processors, and exporters to chain together for that signal.
  • If docker compose up fails, make sure Docker Desktop is running first.
  • If port 4317 or 4318 is already in use, stop any previously running Collector container: docker stop otel-collector.
  • If your app exits without sending data, it may be using SimpleSpanProcessor which exports synchronously — the Collector needs to be running before the app starts.
  • If you see no output in docker logs, check that your app's OTLP endpoint matches — it must be http://localhost:4318/v1/traces (HTTP) or localhost:4317 (gRPC).
  • Inspect detailed Collector startup logs: docker logs otel-collector 2>&1 | head -40
