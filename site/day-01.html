The Signals Matrix

Before we create data, you need to know what data to create. OpenTelemetry categorizes data into three "Signals". Use this matrix to decide what you need:

Signal What is it? Best Used For... Drawback
Traces 🚥 The journey of a single request across multiple services. Debugging latency ("Why is checkout slow?") & finding bottlenecks. High data volume / expensive to store blindly.
Metrics 📊 Numbers aggregated over time (e.g. CPU %, Error rate). Dashboarding & Alerting ("Page me if errors > 5%"). Cannot tell you which user failed, only that failures happened.
Logs 📝 A timestamped record of a specific discrete event. Auditing ("User X changed password") & deep debugging. Lacks context without Trace IDs attached to them.

Module: Manual Instrumentation

These examples show the smallest useful tracing setup: a tracer provider, a span processor, a console exporter, and a few spans.

The Code

Here's the basic payment tracing example that demonstrates nested spans:

python — payment.py 
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import SimpleSpanProcessor, ConsoleSpanExporter

def configure_tracer():
    exporter = ConsoleSpanExporter()
    span_processor = SimpleSpanProcessor(exporter)
    provider = TracerProvider()
    provider.add_span_processor(span_processor)
    trace.set_tracer_provider(provider)
    return trace.get_tracer("payment.py", "0.1.0")

tracer = configure_tracer()

@tracer.start_as_current_span("Starting Payment")
def process_payment():
    print("processing payment")
    validate_card()
    charge_bank()

def validate_card():
    with tracer.start_as_current_span("Validating Card"):
        print("validating card")

def charge_bank():
    with tracer.start_as_current_span("Charging Bank"):
        print("charging bank")

if __name__ == "__main__":
    with tracer.start_as_current_span("Payment Service"):
        process_payment()
🐍 Demystifying the Python "Magic"
You can create spans two ways: Decorators (@tracer...) or Context Managers (with tracer...). What's the difference?
  • Use the @decorator when you want to trace an entire function from start to finish automatically.
  • Use the with block when you only want to trace a specific chunk of code inside a larger function.

Enriching Spans: Attributes, Events, and Status

A bare span just has a name and a duration. The real value comes from enriching it with context. There are three key ways to do this:

🏷️
Span Attributes
Key-value metadata attached to the span — user ID, HTTP method, order amount. Searchable in your backend.
📌
Span Events
Timestamped log entries inside a span — "payment declined", "retry attempt 2". Like a structured log within the trace.
🚦
Span Status
UNSET (default), OK (success), or ERROR (failure). Backends use this to flag failing operations.
python — enriched span example
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import SimpleSpanProcessor, ConsoleSpanExporter

provider = TracerProvider()
provider.add_span_processor(SimpleSpanProcessor(ConsoleSpanExporter()))
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("payment.py", "0.1.0")

def validate_card(card_number):
    with tracer.start_as_current_span("Validating Card") as span:
        # Attributes: searchable key-value metadata on the span
        span.set_attribute("card.last4", card_number[-4:])
        span.set_attribute("card.type", "visa")

        # Event: a timestamped note inside the span
        span.add_event("Card lookup started")

        if card_number == "0000-0000-0000-0000":
            # Status ERROR: marks this span as failed in your backend
            span.set_status(trace.StatusCode.ERROR, "Card declined")
            span.record_exception(ValueError("Invalid card number"))
            return False

        span.add_event("Card validated successfully")
        span.set_status(trace.StatusCode.OK)  # Explicit success
        return True

if __name__ == "__main__":
    with tracer.start_as_current_span("Payment Service"):
        validate_card("4242-4242-4242-4242")
        validate_card("0000-0000-0000-0000")  # This one fails
💡 Status vs Attributes
Status is a first-class field — backends use it to mark spans as failed, trigger alerts, and let you filter for only error spans.

Attributes are queryable metadata — you can filter traces by card.type="visa" or aggregate by http.route.

Events are the timeline inside a span — useful for understanding when something happened during a long operation.
